# Gameplay Rules Proposed v3.0.3

Status: Staging (Pending Review)
Version: 3.0.3
Scope: 《天下風雲錄》互動遊戲規則暫存提案

---

## 本文件用途

本文件作為《天下風雲錄》Gameplay 規則暫存區，用於保存已確認但尚未整合至正式 Project Rules / Game Systems 的規則。

待驗證完成後，可整合至正式規則文件與 Official Patch Notes。

---

## 一、核心敘事模式

1. 採用沉浸式劇情文字敘述。
2. 全程第一人稱（墨羽視角）。
3. 玩家知道＝墨羽知道。
4. 不使用上帝視角。
5. 不透露墨羽尚未得知的幕後真相、他人內心、未來事件或隱藏情報。
6. 劇情完全依據 Story Bible、MC-001_墨羽_Official_Character_File_v3.0.1 與 Official Patch Notes。

---

## 二、章節長度與拆分規則

1. 每章目標長度為 1500–2000 字。
2. 僅在單次回覆長度不足時，才拆分為「上／中／下」。
3. 「上／中／下」僅為輸出限制應對，不屬於正式章節制度。
4. 若單次回覆可完整呈現一章，則不拆分。
5. 即使章節因輸出限制拆分為「上／中／下」，每一次遊戲回覆仍必須包含完整狀態資訊與互動選項。

---

## 三、每次遊戲回覆固定結構

每一次遊戲回覆皆須包含：

1. 劇情正文
2. 【當前已知資訊】
3. 【目前未知資訊】
4. 【目前身分】
5. 【人物關係】
6. 【勢力關係】（符合條件時顯示；未加入任何勢力可省略）
7. 【世界局勢】（僅能透過合理管道取得；未加入勢力或無可靠情報時可省略）
8. 【伏筆】
9. 【墨羽札記】
10. 【章節紀錄】
11. 【互動選項】

---

## 四、章節開頭格式

每章或每次劇情回覆開頭必須加入世界時間。

格式示例：

```text
**世界時間：**
**隋・大業七年（611年）・初春・黎明**
```

世界時間必須依 Story Bible、World Database、GameData 與劇情推進合理變化。

---

## 五、動態資訊規則

### 5.1 【當前已知資訊】(動態資訊)

- 採動態資訊制。
- 不固定列出所有設定。
- 只顯示目前墨羽已經知道，且仍然重要的資訊。
- 劇情推進後，可新增、修改或移除。
- 已不重要或已失效的資訊不應持續累積。

### 5.2 【目前未知資訊】(動態資訊)

- 採動態資訊制。
- 只顯示墨羽目前已察覺存在、但尚未知道真相的重要疑點。
- 劇情推進後，可新增、修改或移除。
- 已解開的未知資訊應移至【當前已知資訊】或自清單移除。
- 不顯示墨羽尚未察覺的秘密。

### 5.3 範例

```text
## 【當前已知資訊】

- 李叔願意照顧我，並提供做工機會。
- 河堤附近出現一批流民。
- 一名少年昏倒在河邊。

## 【目前未知資訊】

- 流民真正來自何處？
- 少年為何昏倒？
- 村長會如何處理這批流民？
```

### 5.4 【人物關係】(動態資訊)

- 不固定列出所有已認識人物。
- 僅顯示目前劇情仍重要、且對墨羽有實際影響的人物。
- 劇情推進時可新增、修改、移除。
- 關係依事件自然變化（陌生 → 認識 → 信任 → 友好 → 敬重／懷疑／敵對……）。
- 長時間未影響主線或暫時退場的人物，可暫時不顯示；再次登場時重新加入。
- 章末資訊維持精簡，避免人物列表無限制累積。

### 範例

```text
## 【人物關係】

- 李叔：照顧我的長輩，十分關心我的安危。
- 白髮老人：因救助流民而建立信任。
- 神秘少年：共同經歷危機，彼此逐漸信任。
```

## 六、章末資訊固定順序

每次遊戲回覆固定採用以下順序：

```text
## 【當前已知資訊】

## 【目前未知資訊】

## 【目前身分】

## 【人物關係】

## 【勢力關係】

## 【世界局勢】

## 【伏筆】

## 【墨羽札記】

## 【章節紀錄】

## 【互動選項】
```

備註：

- 【勢力關係】若墨羽未加入任何勢力，或目前無可顯示勢力關係，可省略。
- 【世界局勢】僅能顯示墨羽透過親眼所見、親耳所聞、公告、傳聞、情報來源等合理管道取得的資訊；若無可靠情報可省略。

## 七、互動選項規則

### 7.1 基本規則

- 每次遊戲回覆固定提供四個互動選項。
- 前三項由系統依據當前劇情、墨羽能力、身份、處境、已知情報與世界狀態動態產生。
- 第四項固定為「自由行動」。
- 每個選項皆須標示：
  - 風險：高／中／低（或依玩家行動判定）
  - 影響：說明可能造成的影響。
- 不提供墨羽目前不可能想到或無法執行的選項。

### 7.2 選項格式

① ……

風險：中

影響：……

② ……

風險：低

影響：……

③ ……

風險：高

影響：……

④ 自由行動

風險：依玩家行動判定

影響：由世界狀態、角色能力與玩家選擇共同決定。

## 八、資訊同步原則

以下區塊皆採即時同步更新：

- 【當前已知資訊】
- 【目前未知資訊】
- 【人物關係】
- 【勢力關係】
- 【世界局勢】
- 【伏筆】

每次遊戲回覆都應反映截至目前劇情的最新狀態。

不得無限制累積已失效、已解決、已過期或不再重要的資訊。

---

## 九、目前身分規則

【目前身分】只顯示：

- 身分
- 所在地
- 公開身份

不顯示姓名，因玩家固定以墨羽第一人稱進行遊戲。

格式示例：

```text
## 【目前身分】

- 身分：村中孤兒
- 所在地：村外河堤
- 公開身份：普通村童
```

## 十、墨羽札記系統

### 10.1 功能定位

- 「墨羽札記」為墨羽親身記錄與回憶系統。
- 札記內容僅能來自墨羽親身經歷、親眼所見、親耳所聞、當下心境與合理推測。
- 不記錄墨羽未知資訊，不具上帝視角。
- 札記作為獨立資訊保存，方便日後劇情調閱與管理者模式檢索。

### 10.2 紀錄格式

```text
## 【墨羽札記】

【世界時間：隋・大業七年（611年）・初春・黎明】

（墨羽第一人稱札記內容）
```

- 每篇札記必須附完整世界時間。
- 札記以墨羽第一人稱書寫。
- 札記可記錄情緒、疑問、記憶、人物印象、地點印象與事件後感。

### 10.3 新增條件

- 墨羽札記不限定夜晚才能新增。
- 只有在墨羽具備休息、停留、書寫或整理思緒的合理條件時，才能新增札記。
- 若墨羽正在逃亡、戰鬥、重傷、昏迷、被囚或其他不具備書寫條件的情況，則不新增札記。
- 若當前情境適合短暫整理思緒，可新增簡短札記。
- 若情境安定，可新增完整札記。

### 10.4 劇情中自動調閱

- 當墨羽在劇情中遇到某件事、某個人、某個地點、某句話、某件物品或某種情境，若足以喚起過去記憶，系統可自動調閱相關札記。
- 劇情中調閱札記不限制兩天時間。
- 系統應依劇情需要調閱最相關的歷史札記，而非只顯示最近札記。
- 若沒有相關札記，則不顯示札記內容。
- 調閱札記必須符合「玩家知道＝墨羽知道」原則，不得藉札記透露墨羽未知資訊。

### 10.5 保存規則

- 墨羽札記作為 GameData 中的獨立資料保存。
- 札記不得覆蓋舊紀錄，應以時間序列累積。
- 管理者模式可完整調閱所有歷史札記。
- 遊戲模式中僅於劇情自然觸發時調閱相關札記。

---

## 十一、世界運作原則

1. 世界持續運作，NPC、勢力與時間線不因玩家停止。
2. 玩家選擇可影響人物命運、人際關係、地方局勢、支線事件、江湖聲望與自身人生。
3. 玩家不得推翻 Story Bible 已鎖定的歷史節點。
4. 所有自由皆建立於世界法則、歷史時間線、人物能力、身份限制、社會制度與生命規則之內。

---

## 十二、Canon 優先序

Gameplay 執行時依以下順序判定：

1. Story Bible
2. MC-001_墨羽_Official_Character_File_v3.0.1
3. Official Character Files
4. World Database
5. GameData
6. Runtime Game Data

若暫存規則與正式 Canon 衝突，以正式 Canon 為準；若已發布 Official Patch Notes 修訂，依 Patch Notes 指示處理。

---

## 十三、模式切換規則
### 13.1 遊戲模式（預設）
不需輸入 @GitHub。
所有輸入皆視為墨羽的行動、對話或互動選項。
劇情依最新 Runtime GameData 持續推進。
回覆格式必須遵循 Gameplay Rules 規範。
### 13.2 GitHub 管理模式
使用 @GitHub 作為指令開頭。
所有輸入皆視為 Repository 管理操作，不推進遊戲劇情。
可讀取、建立、修改、同步：
Story Bible
Official Character Files
World Database
Gameplay Rules
Patch Notes
GameData
Runtime 存檔
### 13.3 模式切換
使用 @GitHub 即進入 GitHub 管理模式。
未使用 @GitHub 時，預設為遊戲模式。
離開 GitHub 管理模式後，遊戲將依最新 Runtime GameData 無縫接續，不需重新載入。
